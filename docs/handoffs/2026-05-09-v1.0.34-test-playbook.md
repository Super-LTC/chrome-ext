# v1.0.34 Test Playbook — for Michael (Windows)

**What ships in v1.0.34:**
1. Auth doesn't logout on transient backend errors (5xx, single 401)
2. ICD-10 viewer reads live PCC dx list from the DOM (no more 24h sync lag)
3. Med Diag CP/Query columns work for codes like G20.B1 / F03.A0 / F32.A

Goal: confirm all three behaviors, and capture diagnostics if anything breaks.

---

## Step 1 — Get v1.0.34 installed

### If auto-updater is set up
1. Wait up to 30 min, or run `update-super-ltc.bat` from the Start menu shortcut to force an update now.
2. Open `chrome://extensions` in Chrome.
3. Find **Super LTC for Point Click Care**.
4. Confirm the version reads **1.0.34**.
5. Click the **Reload** button (circular arrow icon) on the extension card.

### If installing fresh
1. Download `super-ltc-extension.zip` from https://github.com/Superjonathan123/chrome-ext/releases/tag/v1.0.34
2. Unzip it somewhere stable (e.g., `C:\SuperLTC\`).
3. In `chrome://extensions`, enable **Developer mode** (top right toggle).
4. Click **Load unpacked** → select the unzipped folder.
5. Confirm version 1.0.34.

### Verify in Chrome
- Open any PointClickCare page.
- The Super LTC FAB (orange checkmark, bottom-right) should appear.
- Click it → side panel opens.
- If you're not logged in, log in via the panel.

**Stop here and report if any of this failed.**

---

## Step 2 — Verify the auth diagnostic logs are wired up

This proves the new visibility works *before* we try to break anything.

1. In `chrome://extensions`, find Super LTC.
2. Under **Inspect views**, click **service worker**. (DevTools opens.)
3. Switch to the **Console** tab.
4. Leave that window open in the background.
5. In your normal Chrome window, click the FAB → click **Logout** in the side panel (or the user menu, wherever logout is).
6. Look at the service-worker console — you should see:
   ```
   [Auth] Token cleared: { reason: 'explicit-logout', at: <timestamp> }
   ```

If you see that line, the diagnostic logging is working. Log back in for the next steps.

**If you don't see the line** → the new build isn't running. Reload the extension and try again.

---

## Step 3 — Use the extension normally for a few hours

This is the only real test for the auth fix — the bug was logout under transient backend errors, which we can't reliably simulate.

1. Use the extension across the day like you normally would (different patients, different features).
2. **Leave the service-worker DevTools window open in the background.** (You can minimize it.)
3. If you get logged out at any point, **before logging back in**:
   - Bring the service-worker DevTools window forward
   - Look for a `[Auth] Token cleared:` log line
   - **Take a screenshot** of the console (or copy the log text)
   - Then log back in normally

### What the log tells us

| Log says | What it means |
|---|---|
| `reason: 'persistent-401-apiRequest'` | Backend genuinely returned 401 twice for an API call — backend issue, send screenshot |
| `reason: 'persistent-401-validate'` | Backend genuinely returned 401 twice on token validation — backend issue, send screenshot |
| `reason: 'explicit-logout'` | You clicked logout — expected, ignore |
| **No `[Auth] Token cleared:` line at all** | Chrome itself wiped storage (auto-updater, profile issue, permissions change). This is a different bug we'll need to fix separately. **This is the most important case to catch.** |

## Step 4 — Verify ICD-10 live PCC overlay

This is the new feature.

1. Open a patient in PCC.
2. Navigate to their **Medical Diagnosis** page (the dx list).
3. Pick a code that's **on the dx list** but **NOT** something the extension has been suggesting. For example: a recently-added code.
4. Click the purple **AI Code Patient** button.
5. The ICD-10 viewer should open with three panels.
6. In the **Approved** section of the left sidebar, confirm that every code currently on PCC's dx list shows up. Count them — should match the row count in PCC's table.

### Test the post-submit refresh
1. In the viewer, click a code in **Top Picks** that's NOT yet on the dx list.
2. Stage it (the right panel will show buttons).
3. Submit it via "Code patient" (or whatever the batch submit button says).
4. After the submit completes, the just-staged code should **move into Approved** automatically — without you closing and reopening the viewer.
5. Top Picks should no longer show that code.

### Test manual PCC additions
1. Close the viewer.
2. In PCC's dx list, add a code manually using PCC's normal **update** flow (not the AI Code Patient button). Pick something obscure.
3. Reopen the ICD-10 viewer (click AI Code Patient again).
4. The new code should already be in **Approved**, even though the daily sync hasn't run.

**If any of these fail**, open DevTools (F12) on the PCC page, switch to the **Console** tab, and check for errors. Take a screenshot.

## Step 5 — Verify the Med Diag CP/Query columns fix

1. Find a patient with one of these codes (or any ICD-10 code with a letter after the dot):
   - **G20.B1** — Parkinson's with dyskinesia
   - **F03.A0** — Unspecified dementia, mild
   - **F32.A** — Depression, unspecified
2. Go to that patient's Medical Diagnosis page.
3. Look at the **CP** column (green/amber/red shield) for those rows.
4. **Before v1.0.34:** the CP column was empty for those codes.
5. **In v1.0.34:** they should now show a colored shield like every other row.

If a code is still missing its shield:
1. Open DevTools on the PCC page (F12) → Console tab.
2. Look for `[MedDiagAugment] no diagnosis for code:` warnings.
3. Screenshot any matches.

## Reporting

Whether things work or break, send a quick note with:

1. **What version is installed** (from `chrome://extensions`)
2. **Whether each of the 5 steps above worked** (a row of ✓/✗ is fine)
3. **Any screenshots** of console logs (especially `[Auth] Token cleared:` if you got logged out)
4. **What patient / facility you tested on** (so we can correlate with backend logs)

If you got logged out and saw **no** `[Auth] Token cleared:` line, that's the most important finding — it means Chrome wiped storage on its own and we need a different fix.

## Known limitations to ignore

- **Service-worker console may show the worker as "inactive"** sometimes — that's normal. Clicking the link reactivates it.
- **First viewer open after reload may feel slow** — the Preact bundle loads lazily. Subsequent opens are instant.
- **The `[MedDiagAugment]` warning may fire briefly during page load** if the dx table renders before our script finishes scanning — that's fine as long as the chips appear within a second or two.
